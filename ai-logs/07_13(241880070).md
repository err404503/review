这是 Carter et al. [CFG⁺78] 提出的经典技巧：

选择一个通用哈希函数（universal hash function）
h: U → [m]，其中 m ≈ s / εᵢ（s 是当前块的目标大小 ≈ 2ⁱ，εᵢ 是 Bᵢ 的假阳性率）。

查询时：计算 h(x)，检查它是否在字典里。

为什么能近似？

如果 x 在集合中，h(x) 一定在字典里 → 无假阴性。
如果 x 不在集合中，h(x) 碰撞到已有元素的概率 ≈ εᵢ → 假阳性率 ≈ εᵢ。

2. Bᵢ.Insert(x) —— 插入操作
pseudocode复制B_i.Insert(x):
    y = h(x)          // 用固定的 universal hash h 计算
    Dictionary_i.Insert(y)   // 插入到 Raman-Rao 动态字典中

Raman-Rao 动态字典 负责维护这些 hash 值（支持动态插入、删除等）。
时间：expected amortized O(1)（Raman-Rao 保证）。
空间：(1 + o(1)) ⋅ 2ⁱ ⋅ log(1/εᵢ) 比特（接近信息论下界）。

3. Bᵢ.Membership(x) —— 查询操作
pseudocode复制B_i.Membership(x):
    y = h(x)
    if Dictionary_i.Contains(y):
        return Yes
    else:
        return No

时间：worst-case O(1)（Raman-Rao 动态字典支持常数时间查询，高概率）。
这就是为什么整个 warm-up 构造的单个 Bᵢ 查询是常数时间，但整体查询是 O(log n)（要查多个 Bᵢ）。

4. 关键参数

哈希域大小 m ≈ 2ⁱ / εᵢ（与块大小和目标假阳性率相关）。
哈希函数：固定在 Bᵢ 初始化时选好（pairwise independent 或更好）。
字典：使用 Raman and Rao [RR03] 提供的空间最优（up to lower order terms） 的动态字典实现。

Bᵢ 内部本质上就是一个带哈希压缩的动态精确字典。
插入 = 计算哈希 + 字典插入
查询 = 计算哈希 + 字典查询
所有高性能（常数时间 + 接近最优空间）都来自 Raman-Rao 的动态字典实现。

为什么内部查询是常数时间，动态字典本质是bool数组还是别的什么

Grok
很好的问题！ 我来详细解答 Bᵢ 内部的动态字典 为什么能做到查询常数时间，以及它本质上是什么结构。
1. 动态字典本质是什么？（不是简单的 bool 数组）
不是 bool 数组（不像经典 Bloom Filter）。
Raman and Rao [RR03] 的动态字典是一种高度空间优化的动态完美哈希 + 检索结构，其核心特点是：

维护一个精确的集合（exact set），而不是近似。
支持动态插入（以及在某些版本中支持删除）。
空间使用接近信息论下界：对于大小 s 的集合，空间约为 s log(u/s) + O(s) 比特（或在近似成员场景中 s log(1/ε) + o(s)）。
所有操作（查询、插入）都是常数时间（worst-case 或 high probability）。

本质结构（简化版）：

两层哈希（或多层完美哈希）：
第一层：用一个全局哈希函数把元素散列到较小的桶（buckets）。
第二层：每个桶使用完美哈希函数（perfect hashing），保证桶内没有冲突。

存储方式：
不是 bool 数组，而是一个压缩的数组或表，里面存的是哈希值（或签名）或者实际映射后的 key。
使用位压缩（bit packing）、检索数据结构（retrieval structures） 等技巧，把信息压到接近理论下界。

查询流程（这就是常数时间的原因）：
计算 1~2 次哈希函数 → 直接定位到数组中的某个位置（或几个候选位置）。
读取那里的值并比较 → 决定 Yes/No。
整个过程只访问固定次数的内存位置，所以是 O(1) worst-case。


2. 为什么查询是常数时间？

完美哈希（Perfect Hashing）：保证没有冲突，或者冲突很少，能在常数时间内解决。
静态 vs 动态：Raman-Rao 的关键贡献是动态情况下依然保持常数时间操作，同时空间几乎最优。
在 Bᵢ 的场景中，因为我们把原始元素映射到 [m]（m ≈ 2ⁱ / εᵢ）这个较小的域上，字典实际维护的是 h(S) 这个大小为 ≈2ⁱ 的集合，进一步降低了复杂度。

对比经典 Bloom Filter：

Bloom Filter：bool 数组 + k 个哈希 → 查询要读 k 个位（也是常数时间），但空间有 1.44 倍常数损失，且不能动态最优。
这里：精确字典 + 哈希压缩 → 空间更优（1+o(1)），查询仍是常数时间。

3. 在 Bᵢ 中的完整流程回顾
Insert(x) in Bᵢ：

计算 y = h(x)（h 是 universal hash 到 [m]）。
把 y 插入到 Raman-Rao 动态字典中（动态完美哈希 + 更新表）。

Membership(x) in Bᵢ：

计算 y = h(x)。
查询 Raman-Rao 字典是否包含 y（常数时间）。
返回结果。

这就是为什么单个 Bᵢ 的查询是常数时间的根本原因。

总结：

动态字典本质：基于动态完美哈希 + 压缩存储的精确动态集合数据结构（不是 bool 数组）。
常数时间查询：依赖精心设计的哈希函数 + 直接数组访问 + 少量比较。
Raman and Rao 的工作在当时（2003 年）是动态字典领域空间-时间最优的代表之一。

1.2 Related Work 解读（详细解释）
这一节是论文的文献综述部分，主要回顾了近似成员查询（Approximate Membership）领域与本文主题（大小未知的动态集合近似）相关的已有工作，并指出它们的不足，为本文的贡献做铺垫。
主要内容结构：
1. Bloom Filters（布隆过滤器）

Bloom (1970) 提出的经典结构：
支持动态插入（on-line insertions）。
空间使用约为 log e · n log(1/ε) ≈ 1.44 n log(1/ε) bits（比信息论下界高约44%）。
缺点：不支持删除（删除一个元素可能导致假阴性）。

Counting Bloom Filters（计数布隆过滤器）：
支持删除，但空间开销增加 Ω(log log n) 因子。
Cohen 和 Matias (2003) 改进：将额外空间降到 O(n) bits，并推广到近似多重集（approximate multiplicity）。

结论：Bloom 滤波器在空间上接近最优，但本文关注的是“大小未知”场景下的更精细空间使用。

2. Dictionary-based Approximate Membership（基于字典的近似成员查询）

Carter et al. (1978) 的早期技术：
用通用哈希函数 h: U → [n/ε]，维护哈希值多重集 h(S)。
如果用接近信息论最优的空间存储这个多重集，就能达到 n log(1/ε) + O(n) bits。
不需要删除时，只存distinct hash values即可，用更紧凑的动态集合结构（如 [ANS10]）实现常数时间操作。

实用改进：见 [BMP+06] 等。

3. On-line vs Off-line 空间分离（Separation of on-line and off-line space requirements）

Dietzfelbinger & Pagh (2008)、Porat (2009)、Bellazougui & Venturini (2013) 等工作：
在静态（static）场景下，能把空间做到非常接近 n log(1/ε)（甚至 o(n) 额外空间），查询时间常数或稍高。
但 Lovett & Porat (2010) 证明：在动态（dynamic）更新场景下，必须有 Ω(n / log(1/ε)) 的空间开销，即使查询只在所有插入结束后进行。

这说明“流式（stream）”构建近似成员结构比静态情况更难。

4. Dynamic Space Usage（动态空间使用，即本文核心场景）
这是最直接相关的部分，作者指出现有方案不够好：

文献中已有的支持“大小未知”的动态方案（如 [ABP+07, HKL08, GWC+06, GWC+10, WJZ+11, WJZ+13]）：
空间：Ω(n log n) bits（远高于最优）。
查询时间：Ω(log n) 或 Ω(log(1/ε))。

共同思路：维护一系列容量几何增长的近似成员结构（geometric increasing capacities）。
每个结构都有自己的假阳性率 εᵢ。
查询时要查所有现有的结构 → 查询时间 Ω(log n)。
为使总假阳性率 ≤ ε，需要 ∑εᵢ ≤ ε，通常让 εᵢ 随 i 指数下降（如 εᵢ = n^{-Ω(i)}），导致空间爆炸到 Ω(n log n)。


作者评价：这些方案“从算法角度看有些 naive”（somewhat naive），渐近界很差。
5. 其他相关技术

Dynamic Perfect Hashing + Retrieval：
静态情况下常用：完美哈希 + 每个 key 存 log(1/ε) 位签名。
但在在线（on-line）动态场景下，完美哈希/检索本身就需要 Θ(n log log n) 空间（[MPP05]），且现有上界是固定空间，无法实现“随当前大小优雅增长”的目标。


本节核心观点总结：

当集合大小 n 已知 时，空间最优是 Θ(n log(1/ε)) bits，且有高效动态实现。
