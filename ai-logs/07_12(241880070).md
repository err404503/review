每个 B_i 只存它自己块里的元素，空间大约是 2^i × log(1/ε_i)。
因为 ε_i 里有 log(i²) = 2 log i 的额外开销，而 i 最大到 log n，所以总的额外空间是 O(n log log n)。
主项仍然是 n log(1/ε)，和已知 n 时的最优空间几乎一样，只多了 O(n log log n) 的低阶项。

优点和缺点
优点：

实现简单直观。
空间几乎最优（匹配了论文的下界）。
插入很快（期望摊销常数时间）。

缺点：

查询慢：需要 O(log n) 次子查询，不如传统 Bloom Filter 的常数时间高效。

这是论文里的“热身构造”（warm-up），后面第5节会给出一个更复杂的构造，实现常数时间查询，同时保持几乎相同的空间。
总结：这个方案用“多个越来越大的 Bloom Filter（或类似结构）”来适应未知大小，空间开销很小，但牺牲了一些查询速度。非常适合空间敏感但对查询延迟要求不极端的场景。

B_i 的设计详解
在论文 Construction 1（热身构造） 中，B_i 并不是作者重新发明的一个新数据结构，而是直接调用已有的、已知最优的动态近似成员数据结构来实现的。
1. B_i 的核心参数

容量：设计为能容纳 2^i 个元素（因为第 i 个子序列正好有 2^i 个元素）。
假阳性率：ε_i = Θ(ε / i²)（作者通过调整常数使 ∑ ε_i ≤ ε）。
功能要求：
支持动态插入（on-line insertion）
支持成员查询（membership query），无假阴性（false negative）
空间接近最优：(1 + o(1)) · 2^i · log(1/ε_i) 比特


2. B_i 具体如何实现？
论文明确指出：
“We can instantiate the B_i’s, for example, with the dynamic approximate membership data structure resulting from the dynamic dictionary of Raman and Rao [RR03] (via the general dictionary-based methodology described in Section 1.2).”
具体构造方式：

底层使用动态字典（Dynamic Dictionary）
使用 Raman 和 Rao 在 2003 年提出的动态完美哈希或动态字典数据结构（空间接近信息论下界，支持常数时间或期望常数时间操作）。

基于字典的近似成员通用方法（Section 1.2 中提到）：
选择一个通用哈希函数 h: U → [m]，其中 m ≈ 2^i / ε_i（即把元素哈希到大约 2^i / ε_i 个槽）。
把哈希后的值 h(x) 存入一个动态字典中（只存 distinct 的哈希值）。
查询时：计算 h(x)，查字典里是否存在 → 这就构成了一个近似成员结构。
这种方式的空间开销约为 n log(1/ε) 比特（这里 n = 2^i），非常接近最优。


所以 B_i 本质上就是一个：

容量固定为 2^i 的
假阳性率控制在 ε_i 的
动态（支持插入）的
近似成员数据结构（最优空间版本）

3. 为什么这样设计 B_i？

每个 B_i 只负责自己那一块（2^i 个元素），大小精确已知 → 可以直接使用“已知大小 n 时最优”的现有技术。
不同 B_i 之间相互独立，后面的 B_i 容量是前面指数倍增长，符合几何级数。
通过让 ε_i 随 i 增大而减小（1/i²），保证所有 B_i 的假阳性率之和仍然 ≤ ε。

4. 实际效果（回顾）

空间：总空间 = Σ space(B_i) ≈ (1+o(1))n log(1/ε) + O(n log log n)
插入：每个元素只插到一个 B_i 中 → 期望摊销 O(1)
查询：必须查所有已存在的 B_1 到 B_{log n} → O(log n) 时间（这是这个构造的主要缺点）


总结：
B_i 不是一个独立的复杂发明，而是“拿已知最优的动态近似成员结构（基于 Raman-Rao 动态字典），针对容量 2^i 和假阳性率 ε/i² 进行实例化”。
这是典型的“分层 + 复用已有最优组件”的做法，简单有效，但查询时间是 O(log n)。论文后面第5节的 Construction 2 才是真正复杂且巧妙的部分，它只维护一个数据结构，就能做到常数时间查询。
